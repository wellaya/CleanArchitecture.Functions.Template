# ADR 0002: EF Core boundary between shared packages and vertical projects
## Status
Accepted
## Context
Platform.Application (shared NuGet package, consumed by many unrelated verticals) must stay
persistence-agnostic — it should not force every consumer to depend on EF Core.
The per-vertical {Name}.Application project (generated by the template, one instance per vertical)
is not shared across verticals, so it may depend on EF Core directly.
## Decision
- Platform.Application.Common.Models.PaginatedList<T> is a plain data class: Items, PageNumber,
  TotalPages, TotalCount. No EF Core reference, no factory method.
- Each vertical's own Application project (e.g. CleanArchitecture.Functions.Application) references
  EF Core directly, defines IApplicationDbContext with DbSet<T>, and executes paginated queries
  (CountAsync/ToListAsync) inline in its query handlers, constructing PaginatedList<T> via its
  public constructor.
## Consequences
- Platform.Application stays reusable by non-EF verticals.
- Each vertical's query handlers have a few extra lines of pagination code instead of one shared
  helper — an accepted, small cost for correct package boundaries.
